# Стритфайт 98 — расширенный камень-ножницы-бумага PvP

Камень-Ножницы-Бумага-Ящерица-Спок с полосой здоровья, снаряжением и
лутбоксами. Быстрый бой со случайным соперником или своя комната по коду.
Визуальный стиль — панк/VHS 90-х, отдельный от Win98-эстетики Шпиона и
Морского боя.

Отдельный сервис — свой код, своя логика, свой деплой, как и договаривались.

## Механика

- 5 вариантов оружия, каждый бьёт ровно два других и проигрывает двум
  (классическая расширенная схема "Камень-ножницы-бумага-ящерица-Спок").
- У каждого бойца 100 HP. Оба игрока одновременно и втайне выбирают оружие
  каждый раунд; сервер раскрывает выбор только когда оба сделали ход.
  Победитель раунда наносит урон (по умолчанию 10 + бонус от редкости оружия).
- Снаряжение: 4 уровня редкости (Common/Rare/Epic/Legendary), у каждого своя
  прибавка к урону. Лутбоксы стоят 20 жетонов, дают случайное оружие случайной
  редкости; апгрейд применяется, только если выпавшая редкость выше текущей.
- Жетоны: **только для лутбоксов, без вывода** — покупаются в будущем через
  Telegram Stars или зарабатываются за победы/поражения. Реальных денег или
  криптовалюты в игре нет — вагеринг ставками намеренно не реализован
  (см. обсуждение в чате: правила Telegram и правовые риски ставок на
  выигрыш, будь то фиат или крипто).
- Экономика игрока (жетоны, снаряжение, рейтинг, W/L) хранится по playerKey —
  либо id из Telegram, либо сгенерированный и сохранённый в localStorage.
  Это MVP-хранилище в памяти процесса; при масштабировании — вынести в Redis/БД.

## Запуск локально

```bash
npm install
node server.js        # http://localhost:3000
```

## Настройки (переменные окружения)

| Переменная | По умолчанию | Что это |
|---|---|---|
| PORT | 3000 | порт HTTP/WS |
| T_MOVE | 20 | секунд на выбор оружия (иначе выбор случайный) |
| T_REVEAL_PAUSE | 1800 | миллисекунд паузы после раскрытия раунда перед следующим |
| BOT_TOKEN | — | токен бота из BotFather; без него профили доверяют клиенту (как раньше) |
| DATABASE_URL | — | строка подключения к Postgres; без неё прогресс живёт только в памяти процесса |
| PGSSL | on | поставь `off`, если Postgres не требует SSL (например, локальная разработка) |
| LOG_LEVEL | info | уровень логирования pino: trace/debug/info/warn/error/silent |
| MSG_RATE_LIMIT | 30 | максимум WS-сообщений с одного соединения за окно |
| MSG_RATE_WINDOW_MS | 10000 | длина окна для MSG_RATE_LIMIT, мс |
| HTTP_RATE_LIMIT | 120 | максимум HTTP-запросов с одного IP за окно (раздача статики) |
| HTTP_RATE_WINDOW_MS | 60000 | длина окна для HTTP_RATE_LIMIT, мс |
| HEARTBEAT_MS | 30000 | как часто пинговать WS-клиентов для обнаружения оборвавшихся соединений |
| BOT_FILL_ENABLED | on | `off` полностью отключает ботов-соперников (когда наберётся живая аудитория) |
| BOT_FILL_MIN_SEC | 12 | минимум секунд ожидания в очереди перед подключением бота |
| BOT_FILL_MAX_SEC | 20 | максимум секунд ожидания перед подключением бота (разброс маскирует тайминг) |
| BOT_DELAY_MIN_MS | 1500 | минимальная «человеческая» задержка хода бота |
| BOT_DELAY_SPREAD_MS | 3000 | разброс задержки хода бота поверх минимума |

## Профиль игрока и сохранение прогресса

- **Экономика** (жетоны, снаряжение, наёмники, рейтинг) хранится в Postgres одной
  JSONB-колонкой на игрока — таблица создаётся сама при первом запуске.
- **Identity внутри Telegram**: клиент передаёт `initData` — подписанную Telegram
  строку. Сервер проверяет подпись HMAC-SHA256 через `BOT_TOKEN` (алгоритм из
  официальной документации Telegram) и берёт настоящий `user.id` из неё, полностью
  игнорируя то, что клиент прислал в `key`. Это не даёт подделать чужой профиль
  через консоль браузера.
- **Вне Telegram** (обычный веб-доступ или невалидная initData): доверяем
  анонимному ключу из localStorage клиента — как и раньше, без более сильной
  идентификации всё равно неоткуда её взять.
- Без `BOT_TOKEN` или `DATABASE_URL` всё продолжает работать как раньше (в памяти,
  без проверки подписи) — ничего не ломается, если их не задать.

## Бот-соперник (решение «пустого зала»)

Пока живой аудитории мало, «Быстрый бой» подкидывает бота-соперника, если за
`BOT_FILL_MIN_SEC`–`BOT_FILL_MAX_SEC` секунд живой соперник не нашёлся. Сделано
так, чтобы игрок не отличил бота от человека:

- бот — обычный игрок в комнате (со своим ником и профилем под уровень игрока),
  просто без сетевого соединения; протокол для живого игрока идентичен PvP;
- ходит с человеческими задержками (`BOT_DELAY_*`), с неидеальной стратегией
  (иногда рандом, иногда контрит частое оружие игрока), не подсматривая ход;
- живой игрок получает реальные жетоны/рейтинг за победу, как в обычном матче;
- профиль бота нигде не сохраняется, а матч против бота не пишется в историю —
  так в клиент и в БД не утекает ни одного признака бота (проверяется тестом
  `test/bot-opponent.test.js`).

**Важно:** это честно ровно потому, что в игре нет ставок на реальные деньги —
матчи идут за внутриигровые жетоны без вывода. Если когда-нибудь появятся
платежи за исход матча, ботов нужно будет либо убрать, либо раскрывать явно.
Отключается через `BOT_FILL_ENABLED=off`.

## Визуал

- **Шрифты лежат локально** в `public/fonts` (лицензия OFL, свободна в т.ч. для
  коммерции), а не грузятся с Google Fonts CDN: быстрее, не зависит от
  доступности CDN, одинаково у всех. Всего ~60 КБ.
  - Заголовки — **Press Start 2P** (аркадный 8×8, полная кириллица).
  - Текст — **Handjet** (пиксельный, кириллица делалась с проф. консультантом).
  - ⚠️ Не использовать **VT323** (кириллицы нет вообще — русский текст незаметно
    падает в системный моноширинный) и **Pixelify Sans** (открытый баг
    google/fonts#9392: отсутствуют заглавные `О` и `П` — вместо них «тофу»).
  - `-webkit-font-smoothing:antialiased` обязателен: авторы Handjet
    предупреждают об артефактах на малых размерах в Safari/WebKit, а Telegram
    на iOS — это WebKit.
- **Фон** — процедурный пиксельный киберпанк-город (`generateCityBackground`):
  рисуется в 180×320 и растягивается с `image-rendering:pixelated`, поэтому
  пиксели настоящие. Каждый заход генерирует новый: силуэт, окна, вывески,
  «настроение» неба (циан/маджента/янтарь/тил), дождь, луна.
- **Глитч** — редкий (раз в 6–16 с) и короткий: полоса развёртки, дрожание
  карточки, RGB-сплит заголовка. Полностью отключается при
  `prefers-reduced-motion`.
- **Палитра** сбалансирована: доминирует циан, янтарь для акцентов, маджента
  точечно — раньше четыре чистых неона конкурировали за внимание.

### Иконки

Сейчас иконки — свои: пиксель-арт, сгенерированный процедурно (данные `PXICONS`
в `index.html`), плюс механизм `PNG_ICONS` для подмены отдельной иконки готовым
PNG (`public/images/<id>.png`).

Готовые наборы с itch.io и подобных площадок использовать можно, **но у каждого
своя лицензия** — часть бесплатна даже для коммерции, часть требует покупки,
часть прямо запрещает использование в приложениях с монетизацией. Поскольку в
планах Telegram Stars, лицензию нужно проверять перед добавлением любого
стороннего ассета в репозиторий.

## Экономика: валидация и история матчей

- **Санитизация** (`sanitizeEconomy` в `server.js`) — единая точка перед каждым
  сохранением профиля: жетоны/рейтинг/победы/поражения приводятся к целым
  неотрицательным числам, снаряжение с неизвестной редкостью откатывается к
  `common`, экипированный наёмник обязан быть в списке разблокированных (иначе
  сброс на `rookie`), уровень наёмника всегда в диапазоне `[1, MAX_MERC_LEVEL]`.
  Так любая порча данных — баг, гонка состояний, ручное вмешательство в БД —
  не расползается дальше по системе.
- **История матчей** — таблица `matches` в Postgres (победитель, проигравший,
  число раундов, время). Пишется после каждого боя, если БД включена. Клиент
  запрашивает последние 10 через `get_history` и показывает в разделе
  «История матчей» в главном меню.

## Стабильность и надёжность

- **Rate limiting**: свой лимит на WS-сообщения с одного соединения
  (`MSG_RATE_LIMIT` за `MSG_RATE_WINDOW_MS`) и на HTTP-запросы с одного IP
  (`HTTP_RATE_LIMIT`/`HTTP_RATE_WINDOW_MS`) — простой sliding-window в памяти
  (`rateLimit.js`), без внешних зависимостей.
- **Heartbeat**: сервер пингует каждое WS-соединение раз в `HEARTBEAT_MS`;
  если клиент не ответил pong'ом с прошлого пинга — соединение считается
  мёртвым и обрывается сразу, не дожидаясь TCP-таймаута (который может
  занимать минуты и держать комнату "как будто живой").
- **Уборка комнат**: точечная TTL-очистка при обычном дисконнекте плюс
  плановый sweep раз в минуту как страховка — на случай, если по какой-то
  причине комната осталась без активного таймера очистки.
- **Логирование**: структурированные JSON-логи через `pino` (`logger.js`),
  уровень настраивается через `LOG_LEVEL`.

## Деплой на Railway (отдельный сервис)

Тот же процесс, что для Шпиона и Морского боя:
1. Новый репозиторий на GitHub (например `streetfight98-online`) — отдельно от
   остальных игр.
2. Залей содержимое этой папки: `server.js`, `lib.js`, `package.json`,
   `package-lock.json`, `public/`, `test/` (без `node_modules`).
   **Важно**: заливай файлы так, чтобы они были в КОРНЕ репозитория, не
   создавай лишнюю вложенную папку — это была причина прошлых ошибок сборки.
3. Railway: New Project → Deploy from GitHub repo → выбери репозиторий.
4. Если Railway не найдёт entrypoint — Set start command → `node server.js`.
5. Settings → Networking → Generate Domain.

## Подключение к Telegram

Аналогично другим играм: BotFather → `/newapp`, привязать к HTTPS-домену.
